## 낙관적 락과 비관적 락 (Optimistic / Pessimistic Lock)

DB에 동시에 수정이 일어나 충돌이 발생할 수 있다. 이를 막는 두 갈래의 전략이 비관적 락과 낙관적 락이다.

- **비관적 락**: row에 접근할 때 미리 Lock을 걸고, 다른 Lock이 없을 때만 수정을 허용한다. → DB가 제공하는 Lock 이용.
- **낙관적 락**: 수정할 때 "내가 먼저 이 값을 수정했다"고 명시해 다른 사람이 같은 조건으로 못 고치게 한다. → Application Level에서 잡는 Lock.

<br>

#### 비관적 락 (Pessimistic Lock)

----

**트랜잭션이 시작될 때 Shared Lock 또는 Exclusive Lock을 걸고 시작하는 방법.** [[Transaction Isolation Level]] 중 Repeatable Read 또는 Serializable 수준의 격리성을 제공한다.

Shared Lock을 걸면 write를 위해 Exclusive Lock을 얻어야 하는데, 다른 트랜잭션이 Shared Lock을 잡고 있으면 Exclusive Lock을 얻지 못해 업데이트할 수 없다. 수정하려면 해당 트랜잭션을 제외한 모든 트랜잭션이 commit되어야 한다.

```
1. Transaction_1 이 Id 2번 읽음 (name = Karol)
2. Transaction_2 가 Id 2번 읽음 (name = Karol)
3. Transaction_2 가 name을 Karol2로 변경 요청
   → Transaction_1이 Shared Lock을 잡고 있어 Blocking
4. Transaction_1 commit (락 해제)
5. Blocking 됐던 Transaction_2의 update 정상 처리
```

→ **[[Transaction]]을 이용해 충돌을 예방**한다.

<br>

#### 낙관적 락 (Optimistic Lock)

----

**version 같은 별도 컬럼을 추가해 충돌적인 업데이트를 막는다.** DB 기능이 아니라 Application Level에서 잡는 락이다. (version 외에 hashcode, timestamp도 사용)

```
1. A가 Id 2번 읽음 (name = Karol, version = 1)
2. B가 Id 2번 읽음 (name = Karol, version = 1)
3. B가 version=1 row 갱신 → (name = Karol2, version = 2) 성공
4. A가 version=1 row 갱신 시도 → 이미 version이 2라 갱신 실패
```

같은 row에 수정 요청이 둘이어도, 먼저 반영된 쪽이 version을 올리므로 뒤늦은 요청은 반영되지 않는다.

<br>

#### 롤백 (Rollback)

----

테이블이 2개 이상일 때 두 번째 테이블 수정 중 충돌이 나면 앞선 수정에 대한 롤백이 필요하다.

- **비관적 락**: 하나의 트랜잭션으로 묶여 있어, 수정 하나가 실패하면 **DB 단에서 전체 자동 Rollback**.
- **낙관적 락**: 트랜잭션을 잡지 않으므로 **롤백 책임이 Application 단에 있다 → 수동 롤백 구현 필요.**

```sql
-- 비관적 락: 하나의 트랜잭션, 실패 시 자동 전체 롤백
BEGIN TRANSACTION;
  UPDATE anotherTable SET ... WHERE id = 2;
  UPDATE theTable   SET name = 'Karol2' WHERE id = 2;
IF AffectedRows == 1 THEN COMMIT; ELSE ROLLBACK;

-- 낙관적 락: 트랜잭션 없음, version 조건으로 충돌 감지, 실패 시 수동 롤백
UPDATE theTable
   SET val1 = @newVal1, version = version + 1
 WHERE id = 2 AND version = @oldVersion;
IF AffectedRows == 1 THEN {정상 처리} ELSE {수동 롤백 처리};
```

<br>

#### 언제 무엇을 쓰나

----

| | 낙관적 락 | 비관적 락 |
|---|---|---|
| Lock 방식 | Application Level (version 컬럼) | DB Level (Shared/Exclusive Lock) |
| 트랜잭션 | 불필요 | 필요 |
| 성능 | 조회 시 Lock 안 걸어 더 좋음 | 상대적으로 느림 |
| 롤백 | 수동 (까다롭고 update 1회 추가) | 트랜잭션 롤백으로 끝 |
| 적합한 상황 | **충돌이 드문 곳** | **충돌이 잦거나 충돌 비용이 큰 곳** |

- 낙관적 락은 조회 시 Lock이 없고 트랜잭션도 필요 없어, **클라이언트가 정보를 받아 갔다가 나중에 수정 요청**하는 흐름(요청→반환→수정요청→적용)에서도 충돌 감지가 가능하다. 비관적 락은 이 사이 트랜잭션을 유지할 수 없다.
- 단, 낙관적 락의 최대 단점은 **수동 롤백**이다. 충돌이 잦으면 한땀한땀 롤백 + update 추가 비용 때문에 오히려 비관적 락보다 불리할 수 있다.
