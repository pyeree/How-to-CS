# 설계: How-to-CS "오늘의 개념" 데일리 (텔레그램 푸시)

- 날짜: 2026-06-22
- 상태: 설계 승인 대기
- 대상 repo: `pyeree/How-to-CS` (Obsidian CS vault)

## 1. 목적 / 배경

면접 대비 CS vault(How-to-CS)에 **"하루에 하나씩"** 학습 장치를 붙인다.
참고 repo(`eomkyeongmun/ai-cs-daily-study`)는 매일 AI로 *새 노트를 생성*하지만,
How-to-CS는 이미 gyoogle 원본 92개 노트가 다 있고 "생성물 직접편집 금지 · `_inbox` 사람층 ·
`복습일`/`priority`/`status` 복습엔진"이라는 고유 구조를 갖는다.

따라서 이 기능은 **새 노트를 만들지 않는다.** 기존 92개 노트 중 **하루 하나("오늘의 개념")** 를
복습엔진 기준으로 골라, ① 옵시디언 안 섹션과 ② 텔레그램 푸시로 전달한다. **AI 호출 없음.**

### 성공 기준
- 매일 정해진 시각에 텔레그램으로 "오늘의 개념" 1건이 온다.
- 메시지만으로 그 자리에서 30초 복습이 된다(자기완결적 = 액티브 리콜 카드).
- 같은 날에는 항상 같은 개념이 선정된다(날짜 시드 고정 → 옵시디언·텔레그램 일치, 재실행 멱등).
- 기존 빌드 규칙을 어기지 않는다(생성물은 생성기가 만들고, 사람층/`_source` 불가침).

## 2. 비목표 (YAGNI)

- AI(OpenAI/Claude)로 새 개념·요약·면접질문 *생성* — v1 제외(전부 기존 노트 재료 사용).
- 카카오톡/메일 발송 — v1 제외(텔레그램만). 채널은 발송 함수만 갈아끼우면 확장 가능.
- `obsidian://` 딥링크 — v1 제외(텔레그램에서 불안정). GitHub 링크만.
- 사용자별/다중 수신자 — 단일 chat_id.

## 3. 전체 구조

```
[매일 cron] GitHub Actions (.github/workflows/daily.yml)
   └─ python _build/daily.py
        ├─ 1. 8개 카테고리 + _inbox 의 .md frontmatter 읽기 (status/priority/복습일/tags) + 제목
        ├─ 2. "오늘의 개념" 1개 선정 (복습 우선 + 날짜 시드)
        ├─ 3. 00_INDEX/🗓 오늘의 개념.md 생성(렌더)  ← obsidian-git이 pull
        └─ 4. 텔레그램 발송 (Secrets: 토큰 + chat_id) — best-effort
   └─ 변경분 있으면 자동 commit & push
```

- `daily.py`는 `convert.py`/`build_index.py`와 **독립**. `_source/`(gyoogle 클론) 불필요 —
  이미 커밋된 노트만 읽으므로 빠르고, `rmtree` 같은 파괴 동작 없음.
- 의존성 없음: **파이썬 표준 라이브러리만**(frontmatter는 정규식 파싱, 텔레그램은 `urllib`).

### 3.1 컴포넌트

| 컴포넌트 | 종류 | 역할 |
|---|---|---|
| `_build/daily.py` | 신규 스크립트 | 선정 + 노트 렌더 + 텔레그램 발송. `--dry-run` 지원(발송·쓰기 없이 콘솔 출력) |
| `00_INDEX/🗓 오늘의 개념.md` | 생성물 | 오늘 개념 카드(제목·플래그·리콜 안내·요약·관련개념·학습루프 바로가기). 손으로 안 고침 |
| `_build/build_index.py` | 기존 수정 | `🏠 Home`에 "🗓 오늘의 개념" 링크 한 줄 추가(로컬 빌드에서도 노출). daily.py를 호출하진 않음 |
| `.github/workflows/daily.yml` | 신규 | cron + `workflow_dispatch`. daily.py 실행 → 변경 시 commit/push |

> daily.py와 build_index.py의 결합도를 낮춘다: daily.py는 단독 실행 가능하고,
> build_index.py는 "오늘의 개념 노트가 존재하면 Home에서 링크"만 한다(없어도 깨지지 않음).

## 4. 선정 알고리즘 (복습 우선, 날짜 시드)

입력: 8개 카테고리 + `_inbox`의 모든 `.md` (대시보드 FROM 범위와 동일). MOC(`＋ …지도.md`)·
INDEX·템플릿은 제외(개념 노트만 = frontmatter에 `tags`/`status` 보유).

우선순위(위에서부터, 먼저 매칭되는 그룹에서 택1):
1. **복습 도래**: `복습일`이 있고 `복습일 <= 오늘` → 그중 **복습일이 가장 빠른 것**.
2. **빈출 신규**: `priority == 1` & `status == 안함`.
3. **신규**: `status == 안함`.
4. **장기 복습**: 위가 다 비면 `status == 완료` 중 택1.
5. (이론상 전부 비면) "오늘은 후보 없음" 메시지.

**날짜 시드 택1**: 같은 그룹 내 동률·랜덤 선택은 `seed = int(hashlib.sha1(YYYY-MM-DD).hexdigest, 16)`,
`pick = candidates_sorted[seed % len]`. → 같은 날 = 같은 결과(멱등), 날마다 달라짐.
정렬 키는 `(folder, 파일명)` 고정으로 결정성 보장.

## 5. 텔레그램 메시지 (액티브 리콜, MarkdownV2)

```
🗓 오늘의 개념 · {카테고리}  {⭐ if priority==1}{🔁 if 복습도래}

*{제목}*

👉 30초 안에 설명해보세요. 막히면 ↓ 펼치기

||{30초 요약 — 없으면 본문 첫 단락}||

{요약 없을 때만: "ⓘ 아직 30초 요약 없음 → 학습루프 ① 채우기 프롬프트로 작성해보세요"}

📖 노트 열기: {GitHub blob URL}
```

- **요약 출처**: 노트의 `<!-- 🔒 MANUAL:START … END -->` 블록에서 `🎤 면접 30초 요약` 본문 추출.
  비어 있으면 노트 본문 첫 단락으로 대체 + 작성 유도 문구.
- **GitHub 링크**: `https://github.com/pyeree/How-to-CS/blob/main/{경로}` — 경로는 URL 인코딩(한글/공백).
- **MarkdownV2 이스케이프**: 제목·요약의 특수문자(`_ * [ ] ( ) ~ \` > # + - = | { } . !`) 이스케이프 처리.
- 면접 질문 자동생성은 v1 제외(AI 없음). 대신 학습루프 ② "출제 프롬프트" 안내로 대체.

## 7. 에러 처리 / 비밀값

- **Secrets**(GitHub repo Settings): `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`. 레포에 평문 금지.
  로컬 실행 시엔 환경변수 또는 `--dry-run`(발송 생략).
- 텔레그램 발송 실패(네트워크/토큰) → **노트 생성·커밋은 계속**, 실패는 로그만(비치명적).
- 후보 0개 → 크래시 없이 "오늘은 후보 없음" 노트/메시지.
- Windows 콘솔(cp949) 규칙: **stdout 출력에 이모지 금지**(파일 본문 이모지는 OK).
- 멱등성: 하루 여러 번 돌려도 같은 노트 → git diff 없음 → 빈 커밋 안 생김.

## 8. 자동 커밋 트레이드오프 (결정 사항)

매일 `🗓 오늘의 개념.md`가 바뀌므로 **하루 1회 자동 커밋**이 쌓인다(참고 repo와 동일, 사용자 수용).
- 장점: 단일 알고리즘(파이썬) 한 곳, 옵시디언은 pull만.
- 대안(채택 안 함): 옵시디언 섹션을 DataviewJS 동적계산 → 커밋 0이지만 알고리즘이 JS/파이썬 이중화.
- **결정: 자동 커밋 방식 채택.** 커밋 노이즈가 거슬리면 추후 DataviewJS로 전환 가능.

## 9. 최초 1회 셋업 (사용자)

1. 텔레그램 `@BotFather` → 봇 생성 → **봇 토큰** 확보.
2. 봇과 대화 시작(아무 메시지) 후 `getUpdates`로 **chat_id** 확인.
3. GitHub repo → Settings → Secrets → `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` 등록.
4. (옵션) 옵시디언 `obsidian-git` 자동 pull 설정으로 노트 자동 동기화.
5. cron 시각 확정(예: 매일 08:00 KST = `0 23 * * *` UTC).

## 10. 테스트 / 검증

- `python _build/daily.py --dry-run` : 선정 결과 + 메시지 미리보기(발송·쓰기 없음).
- 선정 결정성: 같은 날짜 시드로 두 번 실행 → 동일 결과 단언.
- 우선순위: 더미 frontmatter 케이스(복습도래/빈출안함/안함/완료)별 올바른 그룹 선택 확인.
- 빈 요약 노트 → 본문 첫 단락 대체 + 안내문 포함 확인.
- MarkdownV2 이스케이프: 특수문자 포함 제목으로 발송 포맷 오류 없는지.

## 11. 영향 범위 / 빌드 규칙 준수

- 신규: `_build/daily.py`, `.github/workflows/daily.yml`, (생성물) `00_INDEX/🗓 오늘의 개념.md`.
- 수정: `_build/build_index.py`(Home 링크 1줄), `README.md`/`CLAUDE.md`(데일리 기능 문서화).
- 불가침 준수: `_source/` 손 안 댐, 카테고리 폴더 직접편집 없음, `_inbox`/MANUAL 블록 보존.
- 의존성: 표준 라이브러리만(추가 pip 없음).
